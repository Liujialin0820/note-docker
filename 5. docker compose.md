# Docker Compose



## 一、安装Docker Compose

```shell
# Compose目前已经完全支持Linux、Mac OS和Windows，在我们安装Compose之前，需要先安装Docker。下面我 们以编译好的二进制包方式安装在Linux系统中。 
curl -L https://github.com/docker/compose/releases/download/1.22.0/docker-compose-`uname -s`-`uname -m` -o /usr/local/bin/docker-compose
# 设置文件可执行权限 
chmod +x /usr/local/bin/docker-compose
# 查看版本信息 
docker-compose -version
```

## 二、卸载Docker Compose

```shell
# 二进制包方式安装的，删除二进制文件即可
rm /usr/local/bin/docker-compose
```



## 三、使用 Docker Compose 编排 Nginx + Spring Boot 项目

### 1. 整体架构

```
用户浏览器
    ↓  访问服务器 80 端口
  Nginx 容器                ← 负责接收外部请求、反向代理
    ↓  转发到 app:8080
Spring Boot 容器（app）     ← 负责处理业务逻辑
```

**说明：**
- Nginx 负责接收外部请求
- Spring Boot 负责处理业务逻辑
- 外部只暴露 80 端口
- Spring Boot 的 8080 端口**只在 Docker 内部使用**，外部无法直接访问

---

### 2. 目录结构

```
project/
├── docker-compose.yml
├── backend/
│   ├── Dockerfile
│   └── app.jar
└── nginx/
    └── default.conf
```

---

### 3. Spring Boot 的 Dockerfile

**`backend/Dockerfile`**

```dockerfile
FROM eclipse-temurin:21-jre    # 用 Java 21 运行环境

WORKDIR /app                   # 设置容器内工作目录

COPY app.jar app.jar           # 把 app.jar 复制进容器

EXPOSE 8080                    # 声明容器内部使用 8080 端口

CMD ["java", "-jar", "app.jar"] # 容器启动时执行这条命令
```

**意思是：**
- 用 Java 21 运行环境作为基础
- 把 `app.jar` 放进容器
- 容器启动时执行 `java -jar app.jar`

---

### 4. docker-compose.yml

```yaml
services:

  nginx:
    image: nginx:alpine                                              # 使用官方 nginx 镜像
    container_name: my-nginx
    ports:
      - "80:80"                                                      # 对外暴露 80 端口
    volumes:
      - ./nginx/default.conf:/etc/nginx/conf.d/default.conf:ro      # 挂载 nginx 配置（只读）
    depends_on:
      - app                                                          # 等 app 启动后再启动 nginx
    restart: unless-stopped                                          # 容器异常退出时自动重启

  app:
    build: ./backend             # 自动找 ./backend/Dockerfile 来构建镜像
    container_name: my-springboot-app
    expose:
      - "8080"                   # 只对 Docker 内部网络暴露，外部不可访问
    restart: unless-stopped
```

**重点：**

| 配置项 | 说明 |
|--------|------|
| `ports: "80:80"` | nginx 对外暴露 80，外部可访问 |
| `expose: "8080"` | app 只给 Docker 内部网络访问，外部不能直接访问 |
| `build: ./backend` | docker-compose 自动找该目录的 Dockerfile 构建镜像 |
| `depends_on: app` | nginx 等待 app 容器启动后再启动 |
| `restart: unless-stopped` | 容器崩溃自动重启，手动停止则不重启 |

> **不需要 `links`**，因为 Docker Compose 默认会创建内部网络，
> 服务名（如 `app`）会**自动成为内部域名**，容器之间可以直接通过服务名互相访问。

---

### 5. Nginx 配置

**`nginx/default.conf`**

```nginx
server {
    listen 80;

    location / {
        proxy_pass http://app:8080;                          # 转发给 Spring Boot 容器

        proxy_set_header Host $host;                         # 传递原始 Host 头
        proxy_set_header X-Real-IP $remote_addr;             # 传递用户真实 IP
        proxy_set_header X-Forwarded-For $proxy_add_x_forwarded_for;
        proxy_set_header X-Forwarded-Proto $scheme;
    }
}
```

**关键点：**

`proxy_pass http://app:8080` 中的 `app` **不是 IP 地址**，而是 `docker-compose.yml` 里的**服务名**。

Docker Compose 会自动把 `app` 解析到 Spring Boot 容器的内部 IP，**无需手动填写 IP**。

### 6. 启动项目

在 `project/` 目录下执行：

```shell
docker compose up -d --build
```

**说明：**

| 参数 | 作用 |
|------|------|
| `--build` | 先构建 Spring Boot 镜像，再启动 |
| `-d` | 后台运行，不占用终端 |

```shell
# 查看运行状态
docker compose ps

# 查看实时日志
docker compose logs -f
```

## 为什么需要服务编排？

在微服务架构中，一个应用系统通常包含**若干个微服务**，每个微服务还会部署**多个实例**。

如果每个微服务都手动启停，维护工作量会非常大：

| 手动操作的痛点 | 说明 |
|---------------|------|
| 构建镜像 | 要从 Dockerfile build image，或去 Docker Hub 拉取 image |
| 创建容器 | 要为每个服务单独创建多个 container |
| 管理容器 | 要逐个管理这些 container 的启动、停止、删除 |
| 依赖顺序 | 服务之间有依赖关系，必须按顺序启动（如先启动数据库再启动后端） |

---

## 什么是服务编排？

> **服务编排**：按照一定的业务规则，**批量管理容器**。

```
没有编排：                    有编排（Docker Compose）：

docker run mysql ...          docker compose up
docker run redis ...                ↓
docker run backend ...        所有服务按规则自动启动 ✅
docker run nginx ...
（顺序错了就出问题）
```

## 什么是 Docker Compose？

Docker Compose 是一个**编排多容器分布式部署的工具**，提供命令集管理容器化应用的完整开发周期，包括：

- 服务**构建**
- 服务**启动**
- 服务**停止**

---

## Docker Compose 使用步骤

```
第一步：利用 Dockerfile 定义各服务的运行环境镜像
                    ↓
第二步：使用 docker-compose.yml 定义组成应用的各服务
                    ↓
第三步：运行 docker-compose up 一键启动整个应用
```

### 详细说明

| 步骤 | 文件/命令 | 说明 |
|------|-----------|------|
| ① | `Dockerfile` | 定义每个服务的镜像构建方式 |
| ② | `docker-compose.yml` | 声明所有服务、端口、依赖、网络、挂载等配置 |
| ③ | `docker compose up` | 一条命令启动所有服务 |

---

## docker-compose.yml 示例

以一个典型的**后端 + MySQL + Redis + Nginx** 应用为例：

```yaml
version: '3'
services:

  # 数据库服务
  mysql:
    image: mysql:5.7
    environment:
      MYSQL_ROOT_PASSWORD: root
      MYSQL_DATABASE: mydb
    ports:
      - "3306:3306"
    volumes:
      - ./data/mysql:/var/lib/mysql

  # 缓存服务
  redis:
    image: redis:6
    ports:
      - "6379:6379"

  # 后端应用服务
  backend:
    build: ./backend          # 使用 ./backend/Dockerfile 构建
    ports:
      - "8080:8080"
    depends_on:
      - mysql
      - redis                 # 依赖 mysql 和 redis 先启动

  # 反向代理
  nginx:
    image: nginx:latest
    ports:
      - "80:80"
    depends_on:
      - backend
```

## 常用命令

```bash
# 启动所有服务（后台运行）
docker compose up -d

# 停止所有服务
docker compose down

# 查看所有服务状态
docker compose ps

# 查看某个服务的日志
docker compose logs backend

# 重新构建镜像并启动
docker compose up -d --build

# 只启动某个服务
docker compose up -d mysql
```

---

## 与直接用 Docker 命令的对比

| 对比项 | 手动 docker run | Docker Compose |
|--------|----------------|----------------|
| 启动方式 | 每个容器单独运行命令 | `docker compose up` 一键启动 |
| 依赖管理 | 手动控制启动顺序 | `depends_on` 自动处理 |
| 配置管理 | 命令行参数，难以维护 | yml 文件，清晰可读 |
| 版本管理 | ❌ 无法追踪 | ✅ 可 Git 管理 |
| 团队协作 | 每人手动配置 | 共享同一份 yml 文件 |
| 适用场景 | 单个容器调试 | 多服务应用部署 ✅ |

> **结合 GitHub 使用**：把 `Dockerfile` + `docker-compose.yml` 一起上传到 GitHub，
> 任何人克隆项目后，只需 `docker compose up` 一条命令，整个多服务应用立刻运行。
