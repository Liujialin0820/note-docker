# Docker Hub

## 什么是 Docker Hub？

**Docker Hub** 是 Docker 官方提供的**镜像仓库平台**（https://hub.docker.com），
可以理解为**镜像版的 GitHub** —— GitHub 存代码，Docker Hub 存镜像。

### Docker Hub 能做什么？

| 功能 | 说明 |
|------|------|
| 拉取官方镜像 | `docker pull nginx`、`docker pull mysql` 都是从 Docker Hub 下载 |
| 发布自己的镜像 | 把自己构建的镜像推上去，其他人或服务器可以直接拉取 |
| 公开 / 私有仓库 | 免费账号支持无限公开仓库 + 1 个私有仓库 |
| 版本管理 | 用 tag 管理镜像版本，如 `myapp:1.0`、`myapp:2.0` |

### 快速上手：以 Django 项目为例

#### 项目目录结构

```
my-django-project/
├── docker-compose.yml
├── Dockerfile
├── requirements.txt
├── manage.py
├── mysite/
└── nginx/
    └── default.conf
```

#### 第一步：注册并登录 Docker Hub

```bash
# 注册：https://hub.docker.com

# 登录
docker login
# 输入用户名和密码
```

#### 第二步：构建镜像并推送到 Docker Hub

```bash
cd my-django-project

# 构建镜像
docker build -t django-app .

# 打标签（格式：你的用户名/镜像名:版本）
docker tag django-app yourname/django-app:1.0

# 推送到 Docker Hub
docker push yourname/django-app:1.0
```

#### 第三步：在 EC2 上直接拉取运行

```bash
# 拉取镜像（不需要源码，不需要 Python 环境）
docker pull yourname/django-app:1.0

# 上传 docker-compose.yml 到服务器后，一键启动
docker compose up -d
```

### Django 完整架构示例

#### 整体请求链路

```
用户浏览器
     ↓
Nginx 容器 (:80)      ← 接收外部请求，反向代理
     ↓
Django 容器 (:8000)   ← 处理业务逻辑
     ↓
MySQL 容器 (:3306)    ← 数据库
```

#### Dockerfile（Django）

```dockerfile
FROM python:3.12-slim

WORKDIR /app

COPY requirements.txt .
RUN pip install -r requirements.txt

COPY . .

EXPOSE 8000

CMD ["python", "manage.py", "runserver", "0.0.0.0:8000"]
```

#### docker-compose.yml

```yaml
services:

  nginx:
    image: nginx:alpine
    container_name: my-nginx
    ports:
      - "80:80"
    volumes:
      - ./nginx/default.conf:/etc/nginx/conf.d/default.conf:ro
    depends_on:
      - web
    restart: unless-stopped

  web:
    image: yourname/django-app:1.0    # 从 Docker Hub 拉取已构建好的镜像
    container_name: django-app
    expose:
      - "8000"
    environment:
      DB_HOST: db
      DB_NAME: mydb
      DB_USER: myuser
      DB_PASSWORD: mypassword
    depends_on:
      - db
    restart: unless-stopped

  db:
    image: mysql:8
    container_name: mysql-db
    environment:
      MYSQL_DATABASE: mydb
      MYSQL_USER: myuser
      MYSQL_PASSWORD: mypassword
      MYSQL_ROOT_PASSWORD: rootpassword
    volumes:
      - mysql_data:/var/lib/mysql
    restart: unless-stopped

volumes:
  mysql_data:
```

#### nginx/default.conf

```nginx
server {
    listen 80;

    location / {
        proxy_pass http://web:8000;
        proxy_set_header Host $host;
        proxy_set_header X-Real-IP $remote_addr;
    }
}
```

#### 启动

```bash
docker compose up -d
```

### ❓ 为什么 `docker compose up` 之后项目自动就跑起来了？

这是很多人的疑问，核心原因只有一个：

**镜像里已经提前写好了"启动命令"（CMD）**

```dockerfile
CMD ["python", "manage.py", "runserver", "0.0.0.0:8000"]
```

这行的意思是：**这个容器一启动，就自动执行这条命令。**

所以当你执行 `docker run django-app` 或 `docker compose up` 时，
Docker 会自动：

```
创建容器
    ↓
进入容器内部
    ↓
执行 CMD → python manage.py runserver 0.0.0.0:8000
    ↓
项目跑起来了 ✅
```

传统方式你需要手动输：

```bash
# 安装好环境后，还要自己手动运行
python manage.py runserver
```

Docker 把这一步**提前写进了镜像**，所以每次启动容器都自动执行，无需手动操作。

### Docker 的本质

> **Docker = 定义一个"自动运行某程序的隔离环境"**

| 传统部署 | Docker |
|----------|--------|
| 装 Python | `FROM python:3.12` |
| 装依赖 | `RUN pip install -r requirements.txt` |
| 复制代码 | `COPY . .` |
| 开放端口 | `EXPOSE 8000` |
| 手动运行 | `CMD ["python", "manage.py", "runserver"]` 自动执行 |

Dockerfile 本质上就是：**一份自动化的安装 + 启动脚本**。

**换一台服务器时，不需要：**
- ❌ 安装 Python
- ❌ 安装 MySQL
- ❌ 安装 Nginx
- ❌ 配置环境变量

**只需要：**

```bash
docker compose up -d   # ✅ 整个项目跑起来
```

> Docker 不是虚拟机，它是"一个自动运行某个程序的隔离 Linux 环境"。
> 容器的本质是**一个正在运行的主进程**，主进程（CMD）停止，容器就停止。

## Docker Hub vs GitHub + Docker Compose — 分别什么时候用？

这两种方式都能"让别人运行你的项目"，但目的不同：

### GitHub + Docker Compose（适合开发协作）

```
你写代码 → 推到 GitHub（含 Dockerfile + docker-compose.yml）
                ↓
别人 git clone → docker compose up --build
                ↓
Docker 在对方机器上现场构建镜像 → 运行容器
```

**特点：**
- 存的是**源代码 + 构建脚本**，不是镜像本身
- 每次部署都要重新 `build`，花时间
- 适合**开发阶段**：代码经常变，协作者需要看源码、修改、调试

---

### Docker Hub（适合生产部署）

```
你在本机构建镜像 → docker push 推到 Docker Hub
                ↓
服务器 / EC2 执行 docker pull → 直接运行
                ↓
不需要源代码，不需要构建，秒级启动
```

**特点：**
- 存的是**已编译好的镜像**，拿来即用
- 不需要在服务器上安装编译环境（Java、Node、Python…）
- 适合**生产部署**：不想把源码暴露给服务器，部署快

---

### 对比总结

| 对比项 | GitHub + Docker Compose | Docker Hub |
|--------|------------------------|------------|
| 存储内容 | 源代码 + Dockerfile | 编译好的镜像 |
| 部署时需要 | 源码 + docker build | 只需 docker pull |
| 构建时间 | 每次部署都要 build | 提前 build 好，秒拉 |
| 源码暴露 | 服务器上有源码 | 服务器上没有源码 ✅ |
| 适用场景 | 开发协作、本地调试 | 生产服务器部署 |
| 典型命令 | `docker compose up --build` | `docker pull + docker run` |

---

### 实际工作中的完整流程（两者结合）

```
开发阶段                          发布阶段
────────────────                 ─────────────────────────
GitHub                           Docker Hub
 ├── src/（源代码）                 └── yourname/myapp:1.0（镜像）
 ├── Dockerfile                            ↓
 └── docker-compose.yml          EC2 / 服务器
                                   docker pull yourname/myapp:1.0
                                   docker run -d -p 80:8080 yourname/myapp:1.0
```

> 简单记：**GitHub 给人看代码，Docker Hub 给服务器拿镜像**。
> 开发用 GitHub，部署用 Docker Hub，两者互补不冲突。

## 什么是私有仓库？为什么需要它？

Docker 官方提供了 **Docker Hub**（https://hub.docker.com）用于管理**公共镜像**，
任何人都可以从上面拉取镜像，也可以把自己的镜像推送上去。

但有时候我们需要**私有仓库**：

| 场景 | 原因 |
|------|------|
| 服务器无法访问互联网 | 无法从 Docker Hub 拉取镜像 |
| 镜像包含公司内部代码 | 不希望放到公网上泄露 |
| 团队内部共享镜像 | 需要一个内网可访问的镜像仓库 |
| 加快拉取速度 | 局域网内传输比从公网下载快得多 |

> 简单理解：**私有仓库就是搭建在自己服务器上的"自己的 Docker Hub"**，
> 只有内网或授权的人才能访问。

```
公网 Docker Hub                    私有仓库（自建）
  ↑↓ 所有人可访问                    ↑↓ 只有内网/团队可访问
docker pull nginx              docker pull 192.168.1.100:5000/myapp:1.0
docker push myimage            docker push 192.168.1.100:5000/myapp:1.0
```

---

## 一、搭建私有仓库

Docker 官方提供了一个叫 **`registry`** 的镜像，直接用它就可以搭建私有仓库。

```shell
# 第一步：拉取 registry 镜像
docker pull registry

# 第二步：启动私有仓库容器
docker run -id --name=registry -p 5000:5000 registry
# -i  保持容器运行
# -d  后台运行
# -p 5000:5000  把容器的 5000 端口映射到宿主机的 5000 端口

# 第三步：验证是否搭建成功
# 浏览器访问：http://私有仓库服务器IP:5000/v2/_catalog
# 看到 {"repositories":[]} 表示搭建成功 ✅
```

---

## 二、配置 Docker 信任私有仓库

默认情况下 Docker 只信任 HTTPS 仓库，私有仓库用的是 HTTP，
需要手动告诉 Docker："我信任这个地址"。

```shell
# 编辑 Docker 配置文件
vim /etc/docker/daemon.json
```

添加以下内容（将 IP 替换为你的私有仓库服务器真实 IP）：

```json
{
  "insecure-registries": ["私有仓库服务器IP:5000"]
}
```

```shell
# 重启 Docker 使配置生效
systemctl restart docker

# 重启后 registry 容器也需要重新启动
docker start registry
```

> **为什么要这一步？**
> Docker 默认不信任 HTTP 的仓库地址（认为不安全），
> 加入 `insecure-registries` 后，Docker 才会允许从这个地址 push / pull 镜像。

---

## 三、上传镜像到私有仓库

```shell
# 第一步：给镜像打标签，标注它属于哪个私有仓库
docker tag centos:7 私有仓库服务器IP:5000/centos:7
#          ↑原镜像名    ↑私有仓库地址/镜像名:版本

# 第二步：推送到私有仓库
docker push 私有仓库服务器IP:5000/centos:7

# 验证：浏览器访问查看仓库内容
# http://私有仓库服务器IP:5000/v2/_catalog
# 会看到 {"repositories":["centos"]}
```

> **为什么要先打标签（tag）？**
> Docker 通过镜像名称的**前缀**判断推送到哪个仓库。
> `私有仓库IP:5000/centos:7` 的前缀告诉 Docker：
> "把这个镜像推送到 `私有仓库IP:5000` 这个地址。"

---

## 四、从私有仓库拉取镜像

```shell
# 从私有仓库拉取镜像
docker pull 私有仓库服务器IP:5000/centos:7
```

## 完整流程总结

```
私有仓库服务器                        开发机 / 其他服务器
─────────────────                    ──────────────────────
docker pull registry                 
docker run registry :5000      ←→   # 配置 insecure-registries
                                     docker tag myimage IP:5000/myimage:1.0
                                     docker push IP:5000/myimage:1.0
                                     docker pull IP:5000/myimage:1.0
```

| 操作 | 命令 |
|------|------|
| 搭建仓库 | `docker run -id -p 5000:5000 registry` |
| 信任仓库 | 修改 `daemon.json` → 重启 Docker |
| 打标签 | `docker tag 原镜像 IP:5000/镜像名:版本` |
| 上传镜像 | `docker push IP:5000/镜像名:版本` |
| 下载镜像 | `docker pull IP:5000/镜像名:版本` |
| 查看仓库 | 浏览器访问 `http://IP:5000/v2/_catalog` |
